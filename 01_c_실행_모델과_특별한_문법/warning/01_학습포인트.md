# warning 학습 포인트

대상 파일:

- `01_char_func.c`
- `01_char_func_overflow.c`

## 이 폴더의 핵심 메시지

이 폴더는 **C 문자열 함수가 왜 위험할 수 있는지**를 보여 주기 위한 예제 모음이다.  
핵심 원인은 문자열 함수 자체가 "악하다"가 아니라, **함수가 실제 버퍼 크기를 자동으로 알 수 없다는 점**이다.

즉 문제의 중심은 아래 두 가지다.

- 배열이 함수 인자로 넘어가면 크기 정보가 사라진다.
- C 문자열은 길이 필드가 아니라 `\0` 종료 문자에 의존한다.

---

## 1. `01_char_func.c`가 보여 주는 것

### 핵심 메시지

- 함수 안에서 배열 파라미터는 사실상 포인터처럼 보인다.
- `strcpy`는 목적지 버퍼 크기를 모른다.
- `snprintf`처럼 크기를 함께 받는 함수가 더 안전하다.

### 출력 해석

#### `sizeof(local_buf) in main = 5`

`main` 안의 `local_buf`는 진짜 배열이다.

```c
char local_buf[5] = {0};
```

그래서 `sizeof(local_buf)`는 배열 전체 크기인 5가 나온다.

#### `sizeof(buf) inside function = 8`

`inspect_parameter(char buf[])`의 `buf`는 실제로 함수 안에서 포인터처럼 동작한다.

즉 거의 아래와 같다.

```c
inspect_parameter(char *buf)
```

네 환경이 64비트이므로 포인터 크기인 8바이트가 출력된 것이다.

여기서 중요한 것은:

- 호출자는 배열을 넘겼다.
- 하지만 함수는 실제 배열 크기를 모른다.
- 함수가 보는 것은 시작 주소뿐이다.

#### `small_buf = hell`

`safe_copy_demo`에서:

```c
snprintf(small_buf, sizeof(small_buf), "%s", src);
```

가 실행된다.

- `small_buf` 크기는 5
- C 문자열은 끝에 `\0`가 필요함
- 따라서 실제 문자로는 최대 4개만 안전하게 들어갈 수 있음

그래서 `h`, `e`, `l`, `l`까지 쓰고 마지막 1바이트에 `\0`를 넣어 `hell`이 출력된다.

즉:

- 잘린 것은 맞다
- 하지만 버퍼 범위 안에서 멈췄기 때문에 안전하다

## 2. `01_char_func_overflow.c`가 보여 주는 것

### 핵심 메시지

이 파일은 의도적으로 UB(정의되지 않은 동작)를 일으킨다.

```c
char small_buf[5];
const char *src = "hello";
strcpy(small_buf, src);
```

`"hello"`는 메모리상 실제로 아래 6바이트가 필요하다.

```text
'h' 'e' 'l' 'l' 'o' '\0'
```

그런데 `small_buf`는 5바이트뿐이므로 마지막 `\0`까지 쓰는 순간 버퍼 밖으로 1바이트를 넘겨 쓰게 된다.

이것이 버퍼 오버플로우다.

---

## 3. 왜 오버플로우가 났는데도 `small_buf = hello`가 출력되었는가

이게 가장 중요한 질문이다.

결론부터 말하면:

**잘못된 코드인데도 우연히 그럴듯하게 동작했기 때문이다.**

이건 정상 동작이 아니라 **UB(Undefined Behavior)** 다.

### 무슨 일이 일어났나

`small_buf[5]`는 5바이트만 유효하다.  
그런데 `strcpy`는 6바이트를 쓴다.

즉 대략 이렇게 된다.

```text
small_buf에 들어간 것:
'h' 'e' 'l' 'l' 'o'

그 다음 '\0'은
small_buf 바깥 메모리에 써 버림
```

그런데 실행 당시에는 우연히 바로 뒤 메모리를 덮어써도 즉시 크래시가 나지 않았고, 출력 시점에도 문자열처럼 보였던 것이다.

### 왜 이런 일이 가능한가

C는 이런 상황을 막아 주지 않는다.  
운영체제나 CPU도 "지금 스택의 1바이트를 넘겼네"를 항상 즉시 잡아 주지 않는다.

그래서 다음 중 무엇이든 가능하다.

- 겉보기엔 정상 출력됨
- 다른 변수 값이 망가짐
- 이상한 문자열이 출력됨
- 나중에 갑자기 크래시 남
- 이번엔 되고 다음엔 안 됨

이게 바로 UB의 무서운 점이다.

### 핵심 오해 금지

`hello`가 출력되었다고 해서 코드가 맞다는 뜻이 아니다.  
오히려 **"틀렸는데도 잠깐 멀쩡해 보였다"**가 더 위험하다.

---

## 4. 왜 ASan으로 실행해야 하는가

AddressSanitizer는 이런 "겉보기 정상" 상황을 잡아내기 위해 매우 유용하다.

아래처럼 빌드하면:

```bash
gcc -std=c11 -Wall -Wextra -O1 -g -fsanitize=address 01_char_func_overflow.c -o 01_char_func_overflow
```

실행 시 런타임이 스택/힙 경계를 감시하고, 오버플로우를 발견하면 에러를 보고한다.

즉 ASan은:

- 프로그램이 우연히 계속 실행되더라도
- "여기서 범위를 넘겨 썼다"를 정확히 알려 준다

학습 초반에는 이런 도구를 함께 써야 잘못된 직관을 막을 수 있다.

---

## 5. 왜 `strcpy`는 위험한가

`strcpy(dest, src)`는 `dest`의 크기를 인자로 받지 않는다.

즉 함수는 아래를 알 수 없다.

- `dest`가 실제로 몇 바이트인지
- 남은 공간이 얼마나 되는지
- `src` 전체를 복사해도 안전한지

그래서 `strcpy`는 사용자가 잘못된 버퍼를 주면 그냥 넘겨 쓴다.

함수 입장에서는 포인터만 받았기 때문이다.

---

## 6. 왜 `snprintf`는 상대적으로 안전한가

`snprintf`는 크기를 함께 받는다.

```c
snprintf(buf, sizeof(buf), "%s", src);
```

이 경우 함수는:

- 최대 몇 바이트까지 써도 되는지 알고
- 그 범위를 넘지 않도록 쓴다
- 필요하면 잘라낸다
- 보통 널 종료도 보장하려고 시도한다

그래서 정보가 불완전한 `strcpy`보다 훨씬 안전하다.

---

## 7. 이 예제가 포인터와 함수 개념에 연결되는 이유

이 예제가 어려운 이유는 문자열 함수 자체보다, **배열이 함수 경계를 넘을 때 어떻게 보이는가**가 숨어 있기 때문이다.

핵심 연결은 이렇다.

1. 원래는 배열이 있었다.
2. 함수 인자로 넘기면 배열 크기 정보가 사라진다.
3. 함수는 포인터만 본다.
4. 포인터만으로는 실제 버퍼 크기를 알 수 없다.
5. 그래서 길이를 받지 않는 문자열 함수는 위험하다.

즉 이 문제는 결국 포인터와 함수 인자 모델의 문제다.

## 8. 자주 하는 오해

### 오해 1

"출력이 정상인데 왜 틀렸지?"

정상처럼 보이는 출력은 UB의 흔한 함정이다.

### 오해 2

"배열을 넘겼으니 함수도 배열 크기를 알겠지?"

아니다. 함수는 보통 시작 주소만 안다.

### 오해 3

"문자열 길이는 문자 개수니까 `hello`면 5 아닌가?"

C 문자열은 `\0`가 필요하므로 `"hello"`는 6바이트가 필요하다.
