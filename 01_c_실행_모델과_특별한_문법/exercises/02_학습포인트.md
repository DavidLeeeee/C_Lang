# 02_struct_layout 학습 포인트

대상 파일: `02_struct_layout.c`

## 이 파일이 시사하는 핵심

이 예제의 핵심 메시지는 아래 한 줄이다.

**구조체(`struct`)의 크기는 단순히 멤버 크기를 더한 값이 아니라, 정렬(alignment)과 padding까지 반영한 결과다.**

즉 C에서 구조체는 "필드를 순서대로 붙여 놓은 단순 상자"가 아니다.  
CPU가 다루기 좋은 방식으로 메모리에 배치되기 때문에, **멤버 순서가 바뀌면 크기와 중간 빈칸(padding)이 달라질 수 있다.**

---

## 먼저 꼭 알아야 할 개념

### 1. alignment란?

alignment는 어떤 타입이 메모리에 놓일 때 요구하는 **정렬 기준**이다.

예를 들어 보통 64비트 환경에서는:

- `char`: 1바이트 정렬
- `short`: 2바이트 정렬
- `int`: 4바이트 정렬
- `double`: 8바이트 정렬

즉 `int`는 보통 4의 배수 주소에 놓이길 원하고, `double`은 보통 8의 배수 주소에 놓이길 원한다.

이유는 CPU가 그렇게 접근할 때 더 자연스럽고, 플랫폼에 따라 더 빠르거나 안전하기 때문이다.

---

### 2. padding이란?

padding은 구조체 안에서 **정렬을 맞추기 위해 컴파일러가 자동으로 넣는 빈 공간**이다.

이 빈 공간은 네가 직접 선언하지 않았지만, 메모리 배치를 맞추기 위해 삽입된다.

---

### 3. 왜 구조체 끝에도 padding이 들어가나?

구조체 전체 크기도 보통 그 구조체의 가장 큰 정렬 요구사항에 맞춰진다.

이유는 구조체 배열을 만들 때 다음 원소도 올바르게 정렬되어야 하기 때문이다.

예를 들어 `struct A arr[10];`를 만들었을 때:

- `arr[0]`도 정렬이 맞아야 하고
- `arr[1]`도 정렬이 맞아야 하므로
- 구조체 크기 자체가 정렬 기준에 맞는 배수가 되는 경우가 많다

## 출력 해석

---

## 1. struct A

```c
struct A {
    char c;
    int i;
    char d;
};
```

출력:

```text
struct A: sizeof=12
  offsetof(c) = 0
  offsetof(i) = 4
  offsetof(d) = 8
```

### 메모리 배치

`char c`는 1바이트라 offset 0에 바로 놓인다.

```text
offset 0: c (1바이트)
```

그 다음 `int i`는 보통 4바이트 정렬이 필요하다.  
하지만 현재 다음 위치는 offset 1이다. `int`는 offset 1에 바로 놓이면 정렬이 안 맞는다.

그래서 컴파일러가 **3바이트 padding**을 넣는다.

```text
offset 1~3: padding 3바이트
offset 4~7: i (4바이트)
```

그 다음 `char d`는 1바이트라 offset 8에 놓일 수 있다.

```text
offset 8: d (1바이트)
```

여기까지 실제 사용 바이트는 9바이트처럼 보인다.  
하지만 구조체 전체는 가장 큰 정렬 단위인 4의 배수여야 하므로 끝에 **3바이트 padding**이 추가된다.

```text
offset 9~11: tail padding
```

그래서 최종 크기는 12바이트다.

### 요약

- 중간 padding: 3바이트
- 끝 padding: 3바이트
- 최종 크기: 12바이트

---

## 2. struct B

```c
struct B {
    int i;
    char c;
    char d;
};
```

출력:

```text
struct B: sizeof=8
  offsetof(i) = 0
  offsetof(c) = 4
  offsetof(d) = 5
```

### 메모리 배치

이번에는 `int i`가 맨 앞에 온다.  
처음 시작 주소는 보통 이미 구조체 정렬 기준에 맞춰져 있으므로 `i`는 offset 0에 바로 놓인다.

```text
offset 0~3: i (4바이트)
```

그 다음 `char c`는 offset 4에 들어간다.

```text
offset 4: c
```

그 다음 `char d`는 offset 5에 들어간다.

```text
offset 5: d
```

이제 총 6바이트를 쓴 셈이다.  
하지만 구조체 전체는 여전히 4의 배수 크기를 맞추는 편이 자연스럽기 때문에 끝에 2바이트 padding이 들어간다.

```text
offset 6~7: tail padding
```

그래서 최종 크기는 8바이트다.

### 왜 A보다 작아졌나?

핵심은 `int`를 먼저 배치해서 **중간 padding을 줄였기 때문**이다.

`struct A`는 `char -> int -> char`라서 중간에 3바이트를 낭비했다.  
`struct B`는 `int -> char -> char`라서 중간 padding이 없어졌다.

### 요약

- 중간 padding 없음
- 끝 padding 2바이트
- 최종 크기 8바이트

---

## 3. struct C

```c
struct C {
    char a;
    double b;
    short c;
};
```

출력:

```text
struct C: sizeof=24
  offsetof(a) = 0
  offsetof(b) = 8
  offsetof(c) = 16
```

### 메모리 배치

`char a`는 offset 0에 들어간다.

```text
offset 0: a
```

다음은 `double b`다. 보통 `double`은 8바이트 정렬을 원한다.  
하지만 현재 다음 위치는 offset 1이다. `double`은 여기 들어가면 정렬이 맞지 않는다.

그래서 컴파일러는 offset 8까지 맞추기 위해 **7바이트 padding**을 넣는다.

```text
offset 1~7: padding 7바이트
offset 8~15: b (8바이트)
```

그 다음 `short c`는 보통 2바이트 정렬이면 되므로 offset 16에 들어간다.

```text
offset 16~17: c (2바이트)
```

여기까지 쓰면 총 18바이트처럼 보인다.  
하지만 구조체 전체는 가장 큰 정렬 요구사항인 8의 배수에 맞춰지는 경우가 많다.  
그래서 끝에 **6바이트 padding**이 추가된다.

```text
offset 18~23: tail padding
```

그래서 최종 크기는 24바이트다.

### 요약

- 중간 padding 7바이트
- 끝 padding 6바이트
- 최종 크기 24바이트

## 질문에 대한 직접 답변

### 1. 왜 멤버 순서만 바꿨는데 `sizeof`가 달라질 수 있는가?

멤버마다 요구하는 alignment가 다르기 때문이다.  
큰 정렬이 필요한 타입(`int`, `double`) 앞에 작은 타입(`char`)이 오면 padding이 많이 생길 수 있다.

즉 멤버 순서가 바뀌면:

- 중간 padding 양이 바뀌고
- 끝 padding 양도 바뀌며
- 최종 `sizeof`가 달라진다

---

### 2. padding은 어디에 들어가는가?

padding은 두 군데 들어갈 수 있다.

#### 구조체 중간
어떤 멤버를 올바른 정렬 위치에 두기 위해 앞쪽에 삽입된다.

예:
- `struct A`의 `c`와 `i` 사이
- `struct C`의 `a`와 `b` 사이

#### 구조체 끝
구조체 전체 크기를 정렬 기준의 배수로 맞추기 위해 끝에도 들어간다.

예:
- `struct A`의 끝
- `struct B`의 끝
- `struct C`의 끝

---

### 3. alignment 요구사항이 struct layout에 어떤 영향을 주는가?

영향은 아주 직접적이다.

- 각 멤버가 시작할 수 있는 위치를 제한한다
- 그 제한 때문에 padding이 생긴다
- 그 결과 `offsetof`가 달라진다
- 구조체 전체 크기(`sizeof`)도 달라진다
- 나아가 성능과 ABI에도 영향을 줄 수 있다

즉 alignment는 구조체 메모리 배치를 사실상 지배한다.

---

## 왜 이게 중요한가

이걸 알아야 아래를 이해할 수 있다.

- 왜 멤버 순서만 바꿨는데 메모리 사용량이 줄어드는가
- 왜 네트워크/파일 포맷에 struct를 그대로 저장하면 위험한가
- 왜 다른 컴파일러/플랫폼에서 layout이 달라질 수 있는가
- 왜 ABI와 binary compatibility가 중요한가

즉 구조체는 단순 문법이 아니라 **메모리 표현(data layout)** 문제다.

### 오해 1

"구조체 크기는 멤버 크기를 다 더한 값이다"

아니다. padding 때문에 더 커질 수 있다.

### 오해 2

"padding은 구조체 사이에만 생긴다"

아니다. 멤버 사이에도 생기고, 구조체 끝에도 생긴다.

### 오해 3

"멤버 순서는 의미가 없다"

아니다. 메모리 사용량과 정렬에 큰 영향을 준다.

### 오해 4

"내 컴퓨터에서 이렇게 나왔으니 어디서나 같다"

아니다. 플랫폼, ABI, 컴파일러에 따라 달라질 수 있다.
