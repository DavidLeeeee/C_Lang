# 01 토픽에서 배우는 BEST Code

이 문서는 `01_c_실행_모델과_특별한_문법`에서 배우는 내용을 바탕으로,  
**C에서 더 안전하고, 더 이해하기 쉽고, 더 효율적인 코드를 짜기 위한 기본 원칙**을 정리한 문서다.

여기서 말하는 "BEST"는 무조건 짧고 화려한 코드가 아니라 아래 기준에 가깝다.

- 메모리 모델을 속이지 않는 코드
- 함수 경계에서 정보가 사라지지 않게 쓰는 코드
- 구조체와 데이터 레이아웃을 의식한 코드
- 나중에 디버깅과 유지보수가 쉬운 코드
- 컴파일러/도구와 잘 협력하는 코드

---

## 1. 배열은 배열로, 포인터는 포인터로 구분해서 사고하기

### 원칙

배열과 포인터를 같은 것으로 생각하지 않는다.

### 이유

둘은 비슷하게 사용될 때가 많지만, 메모리 모델상 같은 존재가 아니다.

- 배열은 실제 데이터를 담는 객체다.
- 포인터는 주소값을 저장하는 변수다.
- 배열은 많은 표현식에서 포인터처럼 decay되지만, 항상 포인터는 아니다.

이 차이를 놓치면 다음 실수가 바로 나온다.

- 함수 안에서 `sizeof(param)`로 배열 길이를 구하려고 함
- 버퍼 길이 계산이 틀어짐
- 문자열 처리 버그가 생김

### 확인할 수 있는 코드

- [01_array_pointer.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\01_array_pointer.c)
- [01_학습포인트.md](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\01_학습포인트.md)

### 좋은 습관

- 함수에 배열을 넘길 때는 길이도 같이 넘긴다.
- `sizeof(arr) / sizeof(arr[0])`는 호출자 쪽에서 계산한다.
- 함수 파라미터의 `T arr[]`는 사실상 `T *arr`처럼 동작한다는 점을 항상 의식한다.

### 예시

```c
void print_ints(const int *arr, size_t len) {
    for (size_t i = 0; i < len; ++i) {
        printf("%d\n", arr[i]);
    }
}
```

이 방식이 좋은 이유:

- 함수가 배열 시작 주소와 길이를 둘 다 알 수 있다.
- 호출자가 의도를 분명하게 전달한다.
- 나중에 버퍼 오버런 위험을 줄이기 쉽다.

## 2. 문자열 함수는 "길이 정보"를 잃기 쉽다는 전제로 사용하기

### 원칙

C 문자열 함수는 목적지 버퍼 크기를 자동으로 알 수 없다는 사실을 기본 전제로 둔다.

### 이유

함수는 보통 포인터만 받는다.

- `strcpy(dest, src)`는 `dest`가 실제로 몇 바이트인지 모른다.
- C 문자열은 길이 필드가 아니라 `\0` 종료 문자에 의존한다.
- 그래서 크기 정보가 함수 경계에서 사라지면 위험해진다.

### 확인할 수 있는 코드

- [01_char_func.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\warning\01_char_func.c)
- [01_char_func_overflow.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\warning\01_char_func_overflow.c)
- [warning/01_학습포인트.md](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\warning\01_학습포인트.md)

### 좋은 습관

- 가능하면 길이를 받는 함수 사용
- 목적지 버퍼 크기를 항상 의식
- "문자 개수"와 "널 종료 포함 총 바이트 수"를 구분

### 비교

위험한 코드:

```c
strcpy(buf, src);
```

더 나은 코드:

```c
snprintf(buf, sizeof(buf), "%s", src);
```

### 중요한 교훈

오버플로우가 났는데도 출력이 멀쩡해 보일 수 있다.  
이건 코드가 맞다는 뜻이 아니라, **UB가 우연히 겉으로만 정상처럼 보인 것**이다.

## 3. 구조체 멤버 순서는 메모리 효율과 ABI에 영향을 준다

### 원칙

구조체 멤버를 정의할 때는 타입 크기와 alignment를 의식해서 순서를 잡는다.

### 이유

구조체 크기는 단순한 멤버 크기 합이 아니다.

- 멤버마다 alignment 요구사항이 다르다.
- 그 요구사항을 맞추기 위해 padding이 들어간다.
- 멤버 순서가 다르면 중간 padding과 tail padding 양이 달라진다.

### 확인할 수 있는 코드

- [02_struct_layout.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\02_struct_layout.c)
- [02_학습포인트.md](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\02_학습포인트.md)

### 출력에서 봐야 할 포인트

`struct A`

```text
sizeof=12
offsetof(c)=0
offsetof(i)=4
offsetof(d)=8
```

`struct B`

```text
sizeof=8
offsetof(i)=0
offsetof(c)=4
offsetof(d)=5
```

둘의 차이는:

- `struct A`는 `char -> int -> char`
- `struct B`는 `int -> char -> char`

즉 `int`를 앞에 두어 중간 padding을 줄였기 때문에 `struct B`가 더 작다.

### 좋은 습관

- 큰 정렬 단위를 요구하는 멤버를 앞쪽에 배치하는 것을 먼저 검토
- `char`, `bool`처럼 작은 타입을 뒤로 묶는 구성을 고려
- 실제 `sizeof`, `offsetof`로 확인

### 예시

비효율적일 수 있는 구조:

```c
struct Bad {
    char tag;
    double value;
    short code;
};
```

더 나은 방향:

```c
struct Better {
    double value;
    short code;
    char tag;
};
```

주의:

- 항상 "큰 것부터"가 절대 법칙은 아니다.
- ABI 호환, 의미적 그룹핑, 외부 포맷과의 일치도 고려해야 한다.
- 하지만 **메모리 관점에서 먼저 의심해 볼 기본 습관**으로는 좋다.

## 4. 구조체를 파일 포맷이나 네트워크 포맷으로 그대로 저장하지 않기

### 원칙

`struct`를 메모리에 놓인 그대로 파일이나 네트워크로 저장하는 습관을 피한다.

### 이유

구조체 내부에는 padding이 들어갈 수 있고, 플랫폼/컴파일러/ABI에 따라 레이아웃이 달라질 수 있다.

즉 아래 방식은 위험하다.

```c
fwrite(&obj, sizeof(obj), 1, fp);
```

위험한 이유:

- padding까지 저장될 수 있음
- 다른 플랫폼에서 같은 레이아웃을 보장하지 않음
- endianness 문제도 따로 있음

### 좋은 습관

- 필드 단위로 직렬화
- 타입 크기, 정렬, 바이트 순서를 명시적으로 다룸

## 5. `static`, 스코프, 저장 기간을 모호하게 쓰지 않기

### 원칙

변수와 함수는 "어디서 보이는가"와 "얼마나 오래 사는가"를 구분해서 설계한다.

### 이유

C에서 헷갈리는 것은 이름이 아니라 의미다.

- 지역 변수: 함수 호출 동안만 존재
- `static` 지역 변수: 함수 안에 있지만 프로그램 종료까지 존재
- 파일 스코프 `static`: 같은 파일 내부에서만 보임
- 전역 변수: 프로그램 전체에서 접근 가능할 수 있음

이걸 헷갈리면:

- 수명(lifetime) 버그
- 의도치 않은 공유 상태
- 테스트 어려움
- 링크 충돌 위험

### 확인할 수 있는 코드

- [03_storage_duration.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\03_storage_duration.c)

### 좋은 습관

- 가능한 한 전역 상태를 줄인다.
- 파일 내부에서만 쓸 함수/변수는 `static`으로 제한한다.
- 지역 상태와 영속 상태를 명확히 구분한다.

### 예시

```c
static int cache_hits = 0;

static void record_hit(void) {
    cache_hits++;
}
```

이 코드는 파일 내부에서만 상태를 쓰게 제한한다는 점에서, 무분별한 전역 공개보다 낫다.

## 6. 함수 시그니처에 필요한 정보를 숨기지 않기

### 원칙

함수가 동작하는 데 필요한 정보는 시그니처에 드러내는 쪽이 좋다.

### 이유

C는 많은 것을 자동으로 보존하지 않는다.

예를 들어 배열은 함수 인자로 넘기면 길이를 잃는다.  
그러므로 함수 시그니처에서 그 정보를 명시해야 한다.

### 좋은 습관

모호한 함수:

```c
void process(int *data);
```

더 나은 함수:

```c
void process(const int *data, size_t len);
```

이유:

- 읽는 사람도 입력 계약을 이해하기 쉽다.
- 함수 내부에서 경계 체크를 하기가 쉬워진다.
- 디버깅할 때도 전제가 분명하다.

## 7. `const`를 적극적으로 사용해서 의도를 드러내기

### 원칙

읽기 전용 데이터는 가능한 한 `const`로 명시한다.

### 이유

`const`는 단순 장식이 아니라 함수 계약을 더 분명하게 해 준다.

예:

```c
void print_ints(const int *arr, size_t len);
```

이 함수는:

- 데이터를 읽기만 한다는 의도를 드러내고
- 실수로 수정하는 코드를 컴파일 단계에서 막아 준다

### 01과의 연결

포인터가 어려운 이유 중 하나는 "이 포인터로 값을 바꾸는가, 읽기만 하는가"가 불분명할 때다.  
`const`는 이 모호함을 줄이는 좋은 도구다.

## 9. 경고를 무시하지 않기

### 원칙

경고는 귀찮은 소음이 아니라, C에서 중요한 학습 재료다.

### 이유

네가 이미 본 경고:

- 배열 파라미터에 `sizeof`를 적용한 경고

이건 단순히 "컴파일러가 엄격하다"는 뜻이 아니라,

- 함수 인자로 들어간 배열은 실제 배열이 아님
- 길이 정보가 사라졌음
- 그러니 실수하기 쉬움

을 알려 주는 매우 좋은 신호다.

### 좋은 습관

- `-Wall -Wextra`를 기본으로 사용
- 경고를 보면 "왜 위험한가?"를 먼저 확인
- 가능한 한 경고 없는 상태를 목표로 함

## 10. 01 토픽에서 현재까지 찾은 BEST Code 원칙 요약

### 데이터와 주소를 구분한다

- 배열은 데이터 객체
- 포인터는 주소값

### 함수 경계에서 사라지는 정보를 보완한다

- 배열 길이는 같이 넘긴다
- 문자열 버퍼 크기도 같이 의식한다

### 메모리 레이아웃을 고려한다

- 구조체 멤버 순서를 검토한다
- `sizeof`, `offsetof`로 실제 배치를 확인한다

### 수명과 가시성을 분명히 한다

- `static`의 의미를 정확히 구분한다
- 전역 상태를 무분별하게 늘리지 않는다

### 도구와 협력한다

- 경고를 켠다
- 출력으로 검증한다
- UB를 겉보기 정상과 혼동하지 않는다

- C는 선언이 복잡해질수록 오해가 커짐

### 5. `typedef`는 복잡한 타입을 숨길 때만 절제해서 사용하기

이유:

- 가독성을 높일 수도 있지만, 포인터 정체를 가릴 수도 있음

## 01을 끝낼 때 목표로 삼을 태도

좋은 C 코드는 "짧게 쓰는 코드"가 아니라 아래에 가깝다.

- 메모리 관점이 분명한 코드
- 함수 계약이 드러나는 코드
- 데이터 레이아웃을 무시하지 않는 코드
- 우연히 돌아가는 것이 아니라, 왜 돌아가는지 설명할 수 있는 코드

즉 01에서 배우는 BEST Code의 핵심은 문법 트릭이 아니라  
**메모리, 수명, 경계, 레이아웃을 의식하는 습관**이다.
