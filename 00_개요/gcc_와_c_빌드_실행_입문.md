# GCC와 C 빌드/실행 입문

이 문서는 `01`에 들어가기 전에 알아두면 좋은 **C 프로그램이 만들어지고 실행되는 전체 흐름**을 정리한 문서다.  
목표는 아래 4가지를 확실히 잡는 것이다.

- `gcc`가 정확히 무엇인지 이해한다.
- C 소스 파일이 실행 파일이 되기까지의 단계를 이해한다.
- C 프로그램을 어떻게 빌드하고 배포하는지 큰 흐름을 이해한다.
- `gcc` 명령어의 주요 옵션이 무엇을 의미하는지 이해한다.

---

## 1. `gcc`는 무엇인가

`gcc`는 GNU Compiler Collection의 명령어다. 흔히 "C 컴파일러"라고 부르지만, 정확히 말하면 여러 언어용 컴파일 도구 모음의 드라이버(driver)에 가깝다.

네가 아래처럼 입력하면:

```bash
gcc hello.c -o hello
```

`gcc`는 내부적으로 여러 단계를 알아서 호출한다.

- 전처리기(preprocessor)
- 컴파일러(compiler)
- 어셈블러(assembler)
- 링커(linker)

즉, `gcc`는 "소스 파일을 받아 실행 파일을 만들기 위해 필요한 하위 도구들을 조율하는 진입점"으로 보는 게 맞다.

---

## 2. C 파일은 왜 바로 실행되지 않는가

C 파일(`.c`)은 사람이 읽는 **소스 코드**다. CPU는 이 파일을 직접 실행할 수 없다.

CPU가 실제로 실행할 수 있는 것은 결국 기계어(machine code)이며, 운영체제는 보통 그 기계어가 담긴 **실행 파일**을 로드해서 실행한다.

즉 흐름은 이렇다.

```text
.c 소스 코드
-> 전처리
-> 컴파일
-> 어셈블
-> 오브젝트 파일(.o)
-> 링크
-> 실행 파일
-> 운영체제가 로드 후 실행
```

Rust도 사실 크게 다르지 않다. Rust는 `cargo build`가 이 과정을 훨씬 더 잘 감싸 주기 때문에 "바이너리가 툭 떨어지는 느낌"이 강한 것이다. C는 이 단계를 더 직접 보게 된다.

---

## 3. C 빌드 과정 전체 보기

예를 들어 `hello.c`가 있다고 하자.

```c
#include <stdio.h>

int main(void) {
    printf("Hello, world!\n");
    return 0;
}
```

### 3.1 전처리(preprocessing)

전처리기는 `#include`, `#define`, `#if` 같은 전처리 지시문을 먼저 처리한다.

예:

```bash
gcc -E hello.c -o hello.i
```

결과:

- `hello.i`는 전처리 결과 파일이다.
- `#include <stdio.h>`의 내용이 실제로 펼쳐진다.
- 매크로도 이 단계에서 치환된다.

이 단계는 "컴파일 전에 소스 코드를 변형하는 단계"다.

### 3.2 컴파일(compilation)

전처리된 C 코드를 어셈블리 코드로 바꾼다.

```bash
gcc -S hello.c -o hello.s
```

결과:

- `hello.s`는 어셈블리 파일이다.
- 사람이 어느 정도 읽을 수 있다.
- 이 단계에서 최적화가 본격적으로 들어간다.

### 3.3 어셈블(assembly)

어셈블리 코드를 기계어가 담긴 오브젝트 파일로 바꾼다.

```bash
gcc -c hello.c -o hello.o
```

결과:

- `hello.o`는 오브젝트 파일이다.
- 아직 실행 파일은 아니다.
- 함수 기계어, 심볼 정보, 재배치(relocation) 정보 등이 들어 있다.

### 3.4 링크(linking)

여러 오브젝트 파일과 라이브러리를 묶어서 최종 실행 파일을 만든다.

```bash
gcc hello.o -o hello
```

결과:

- `hello`는 최종 실행 파일이다.
- `printf` 같은 외부 심볼이 C 표준 라이브러리와 연결된다.

---

## 4. 가장 자주 쓰는 형태

보통은 전체 단계를 `gcc`가 한 번에 처리하게 둔다.

```bash
gcc hello.c -o hello
```

이 명령은 사실상 아래를 한 번에 하는 것이다.

- 전처리
- 컴파일
- 어셈블
- 링크

그래서 초반에는 이 한 줄을 먼저 익히는 게 좋다.

---

## 5. 실행은 어떻게 되는가

빌드가 끝나면 보통 Linux/WSL에서는 실행 파일이 현재 디렉터리에 생긴다.

```bash
./hello
```

여기서 `./`가 중요한 이유:

- 현재 디렉터리의 실행 파일을 직접 지정한다.
- 리눅스 셸은 기본적으로 현재 디렉터리를 자동 검색하지 않는다.

즉 `hello`라고만 치면 보통 안 되고, `./hello`라고 해야 한다.

운영체제는 이 실행 파일을 받아:

- ELF 형식인지 확인하고
- 필요한 라이브러리를 로드하고
- 프로세스를 만들고
- `main`으로 바로 점프하는 것이 아니라 런타임 시작 코드(startup code)를 거쳐
- 결국 `main`을 호출한다.

중요한 점은 **운영체제가 바로 네 C 소스 파일을 실행하는 것이 아니라, 이미 빌드된 실행 파일을 메모리에 올려 실행한다**는 것이다.

---

## 6. 오브젝트 파일과 실행 파일은 무엇이 다른가

### 오브젝트 파일 (`.o`)

- 아직 덜 완성된 기계어 조각
- 다른 파일과 합쳐질 수 있음
- 외부 심볼이 아직 해결되지 않았을 수 있음
- 보통 직접 실행 불가

### 실행 파일

- 링킹까지 끝난 결과물
- 운영체제가 로드 가능
- 프로그램 시작에 필요한 형식과 메타데이터 포함

쉽게 말하면:

- `.o`는 부품
- 실행 파일은 완성품

## 7. 여러 파일로 이루어진 프로그램은 어떻게 빌드하나

예를 들어 아래처럼 파일이 나뉜 경우를 보자.

- `main.c`
- `math_utils.c`
- `math_utils.h`

보통은 이렇게 빌드할 수 있다.

```bash
gcc -c main.c -o main.o
gcc -c math_utils.c -o math_utils.o
gcc main.o math_utils.o -o app
```

또는 한 번에:

```bash
gcc main.c math_utils.c -o app
```

작은 프로젝트에서는 한 번에 빌드해도 되지만, 파일이 많아지면 보통 `.o` 단위 빌드와 `make`, `CMake`로 넘어간다.

---

## 8. 배포 관점에서 보면 무엇이 생기나

C 프로그램을 배포할 때는 보통 아래 중 하나다.

### 8.1 실행 파일 자체를 배포

- Linux라면 ELF 실행 파일
- Windows라면 `.exe`

가장 직관적이다.

### 8.2 동적 라이브러리와 함께 배포

프로그램이 공유 라이브러리(`.so`, Windows에서는 `.dll`)에 의존하면, 그 라이브러리도 함께 고려해야 한다.

예:

- 실행 파일은 작아질 수 있음
- 라이브러리 업데이트가 쉬울 수 있음
- 대신 실행 환경 의존성이 생김

### 8.3 정적 링크된 바이너리 배포

가능한 라이브러리를 실행 파일 안에 묶어 배포하는 방법이다.

예:

- 배포는 편할 수 있음
- 파일 크기는 커질 수 있음
- 라이선스, libc 차이, 플랫폼 호환성 문제를 따져야 함

초반에는 일단 **"소스 -> 실행 파일"** 흐름과, **"라이브러리가 링크될 수 있다"** 정도까지만 명확히 잡으면 충분하다.

## 9. `gcc` 명령어의 기본 구조

가장 일반적인 모양은 이렇다.

```bash
gcc [옵션들] 입력파일들 -o 출력파일
```

예:

```bash
gcc -std=c11 -Wall -Wextra -O2 hello.c -o hello
```

이걸 해석하면:

- `gcc`: GCC 드라이버 실행
- `-std=c11`: C11 표준으로 컴파일
- `-Wall`: 자주 중요한 경고 켜기
- `-Wextra`: 추가 경고 더 켜기
- `-O2`: 릴리스 수준 최적화
- `hello.c`: 입력 소스 파일
- `-o hello`: 결과 파일 이름을 `hello`로 지정

## 10. 꼭 알아야 할 주요 옵션

### `-o <파일명>`

출력 파일 이름 지정.

```bash
gcc hello.c -o hello
```

이게 없으면 기본 출력 파일명은 보통 `a.out`이다.

### `-std=c11`, `-std=c17`

어느 C 표준 문법 기준으로 컴파일할지 정한다.

```bash
gcc -std=c11 hello.c -o hello
```

초반 학습은 `c11` 또는 `c17`이면 충분하다.

### `-Wall`

기본적으로 중요한 경고를 켠다. 모든 경고는 아니지만 매우 중요하다.

### `-Wextra`

`-Wall`보다 더 많은 경고를 켠다.

초보일수록 경고를 적극적으로 켜는 습관이 중요하다.

### `-Werror`

경고를 에러로 취급한다.

```bash
gcc -Wall -Wextra -Werror hello.c -o hello
```

학습 초반에는 다소 빡셀 수 있지만, 나중에는 품질 관리에 매우 좋다.

### `-g`

디버깅 정보를 넣는다.

```bash
gcc -g hello.c -o hello
```

이 옵션이 있어야 GDB에서 소스 줄 번호 기준으로 보기 훨씬 좋다.

### `-O0`, `-O1`, `-O2`, `-O3`

최적화 수준을 정한다.

- `-O0`: 최적화 없음, 디버깅용
- `-O1`: 약한 최적화
- `-O2`: 일반적인 릴리스 최적화
- `-O3`: 더 공격적 최적화

학습 초기에는 주로 아래 셋을 쓴다.

- Debug: `-O0 -g`
- Release: `-O2`
- Sanitizer 실험: `-O1 -g -fsanitize=...`

### `-c`

링크하지 않고 오브젝트 파일까지만 만든다.

```bash
gcc -c hello.c -o hello.o
```

### `-E`

전처리 결과만 본다.

```bash
gcc -E hello.c -o hello.i
```

### `-S`

어셈블리 파일까지만 만든다.

```bash
gcc -S hello.c -o hello.s
```

### `-I<경로>`

헤더 파일 탐색 경로를 추가한다.

```bash
gcc -Iinclude main.c -o app
```

### `-L<경로>`

라이브러리 탐색 경로를 추가한다.

```bash
gcc main.c -L./lib -lmylib -o app
```

### `-l<이름>`

라이브러리를 링크한다.

예를 들어 `-lm`은 수학 라이브러리다.

```bash
gcc main.c -lm -o app
```

### `-fsanitize=address`

AddressSanitizer를 켠다.

```bash
gcc -g -fsanitize=address hello.c -o hello
```

메모리 오류를 잡을 때 매우 중요하다.

### `-fsanitize=undefined`

UBSan을 켠다.

```bash
gcc -g -fsanitize=undefined hello.c -o hello
```

### `-pthread`

스레드 관련 컴파일/링크 설정을 함께 켠다.

```bash
gcc thread_example.c -pthread -o thread_example
```

## 11. 초반에 가장 자주 쓰게 될 빌드 패턴

### 디버그 빌드

```bash
gcc -std=c11 -Wall -Wextra -O0 -g hello.c -o hello
```

의미:

- 문법 기준 명시
- 경고 많이 켜기
- 최적화 끄기
- 디버깅 정보 포함

### 릴리스 빌드

```bash
gcc -std=c11 -Wall -Wextra -O2 hello.c -o hello
```

의미:

- 일반적인 실행 성능용 빌드

### ASan 빌드

```bash
gcc -std=c11 -Wall -Wextra -O1 -g -fsanitize=address hello.c -o hello
```

의미:

- 메모리 오류 탐지용

## 12. 왜 명령어가 복잡해 보이는가

C의 빌드 명령어가 복잡해 보이는 이유는, 사실 프로그램이 만들어지는 과정의 실제 조각들이 옵션으로 드러나기 때문이다.

Rust에서는 이 복잡도를 주로 `cargo`가 흡수한다.

C에서도 결국 다음 단계로 가면 직접 긴 명령어를 덜 치게 된다.

- `Makefile`
- `CMake`
- 빌드 스크립트

즉, 지금 `gcc` 옵션을 배우는 이유는 매번 길게 치기 위해서가 아니라, **빌드 시스템이 내부에서 뭘 하는지 알기 위해서**다.

## 13. 실전에서 보게 될 대표 흐름

### 가장 단순한 단일 파일 프로그램

```bash
gcc main.c -o app
./app
```

### 경고와 디버깅을 포함한 단일 파일 빌드

```bash
gcc -std=c11 -Wall -Wextra -O0 -g main.c -o app
./app
```

### 여러 파일 프로그램

```bash
gcc -std=c11 -Wall -Wextra -O0 -g main.c util.c parser.c -o app
./app
```

### 오브젝트 파일로 나눠 빌드

```bash
gcc -std=c11 -Wall -Wextra -O0 -g -c main.c -o main.o
gcc -std=c11 -Wall -Wextra -O0 -g -c util.c -o util.o
gcc main.o util.o -o app
./app
```

## 14. 실행 파일은 내부적으로 어떻게 시작되나

초보 단계에서는 아래 정도만 정확히 알면 충분하다.

- 사용자가 실행 파일을 실행한다.
- 셸이 운영체제에 실행을 요청한다.
- 운영체제가 새 프로세스를 만든다.
- 실행 파일과 필요한 라이브러리를 메모리에 매핑한다.
- 런타임 시작 코드가 먼저 실행된다.
- 그 다음 `main`이 호출된다.
- `main`이 끝나면 종료 코드가 운영체제에 반환된다.

즉, 네가 작성한 `main`은 실제로 프로그램의 절대 첫 번째 코드가 아닐 수 있다. 이건 나중에 시스템 프로그래밍과 링킹을 배울 때 아주 중요해진다.

---

## 15. 지금 당장 기억할 최소 핵심

아직 복잡하게 느껴지면 이것만 먼저 기억해도 된다.

1. C 소스 파일은 바로 실행되지 않는다.
2. `gcc`는 소스 코드를 실행 파일로 만들기 위한 여러 단계를 묶어 주는 명령이다.
3. 가장 기본 명령은 `gcc 파일.c -o 실행파일`이다.
4. 실행은 보통 `./실행파일`이다.
5. 자주 쓰는 옵션은 `-std`, `-Wall`, `-Wextra`, `-g`, `-O2`, `-c`다.
6. 나중에는 `Makefile`이나 `CMake`가 이 과정을 자동화한다.

## 16. 입문용 추천 실습

`00`을 읽고 바로 아래 실습을 해 보면 좋다.

### 실습 A

아주 작은 `hello.c`를 만들고 아래를 각각 해 본다.

```bash
gcc -E hello.c -o hello.i
gcc -S hello.c -o hello.s
gcc -c hello.c -o hello.o
gcc hello.c -o hello
./hello
```

목표:

- `.i`, `.s`, `.o`, 실행 파일이 각각 무엇인지 눈으로 확인하기

### 실습 B

아래 두 빌드를 모두 해 본다.

```bash
gcc -std=c11 -Wall -Wextra -O0 -g hello.c -o hello_debug
gcc -std=c11 -Wall -Wextra -O2 hello.c -o hello_release
```

목표:

- 디버그 빌드와 릴리스 빌드가 다르다는 감각 갖기

### 실습 C

파일을 둘로 나눈다.

- `main.c`
