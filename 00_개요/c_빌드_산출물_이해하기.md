# C 빌드 산출물 이해하기

이 문서는 C 프로그램이 빌드되는 과정에서 나오는 각 단계의 산출물이 **정확히 무엇인지** 이해하기 위한 문서다.  
개발을 해 왔더라도 이 부분은 언어와 빌드 도구가 숨겨 버리는 경우가 많아서, 처음 다시 잡는 것이 전혀 이상하지 않다.

목표는 아래를 분명히 구분하는 것이다.

- C 소스 코드 (`.c`)
- 전처리 결과 (`.i`)
- 어셈블리 파일 (`.s`)
- 오브젝트 파일 (`.o`)
- 최종 링크 결과물(실행 파일, 공유 라이브러리 등)

---

## 전체 그림

C 프로그램이 실행 파일이 되기까지의 큰 흐름은 이렇다.

```text
소스 코드(.c)
-> 전처리 결과(.i)
-> 어셈블리(.s)
-> 오브젝트 파일(.o)
-> 링크 결과(실행 파일 또는 라이브러리)
```

각 단계는 "같은 프로그램을 다른 표현 수준으로 바꾼 것"이라고 보면 된다.  
앞 단계는 사람이 더 읽기 쉽고, 뒤 단계로 갈수록 CPU와 운영체제에 가까워진다.

---

## 1. C 소스 코드 (`.c`)

### 이것은 무엇인가

사람이 작성하는 원본 코드다.

예:

```c
#include <stdio.h>

int add(int a, int b) {
    return a + b;
}

int main(void) {
    printf("%d\n", add(2, 3));
    return 0;
}
```

### 이 단계의 특징

- 사람이 읽고 수정하기 쉽다.
- 변수명, 함수명, 타입, 제어문, 매크로 사용 등이 모두 드러난다.
- 아직 CPU가 직접 실행할 수 있는 형태는 아니다.
- `#include`, `#define` 같은 전처리 지시문이 남아 있다.

### 왜 중요한가

이 단계는 사람이 의도를 표현하는 수준이다. 하지만 실제 실행을 이해하려면 이 코드가 이후 단계에서 어떤 모습으로 바뀌는지 봐야 한다.

---

## 2. 전처리 결과 (`.i`)

### 이것은 무엇인가

전처리기(preprocessor)가 처리한 뒤의 C 코드다.

생성 예:

```bash
gcc -E hello.c -o hello.i
```

### 전처리기에서 하는 일

대표적으로 아래를 처리한다.

- `#include`: 헤더 파일 내용을 펼친다.
- `#define`: 매크로를 치환한다.
- `#if`, `#ifdef`, `#ifndef`: 조건부 컴파일을 처리한다.
- 주석 제거 같은 초기 텍스트 정리도 수행한다.

### 이 단계의 특징

- 아직 C 코드다.
- 다만 사람이 직접 쓴 코드보다 훨씬 길어질 수 있다.
- 헤더 파일 내용이 펼쳐져 들어간다.
- 매크로가 실제 치환된 결과를 볼 수 있다.

### 왜 존재하는가

C는 언어 차원에서 전처리기를 강하게 사용한다. 따라서 실제 컴파일러가 보는 코드는 네가 작성한 `.c` 그대로가 아니라, 전처리 결과다.

### 이 파일을 보는 이유

다음 같은 상황에서 특히 중요하다.

- 매크로가 예상과 다르게 동작할 때
- `#include`로 어떤 선언이 들어왔는지 확인하고 싶을 때
- 조건부 컴파일이 실제로 어떤 코드를 남겼는지 보고 싶을 때

### 직관적으로 이해하면

`.i`는 "전처리까지 끝난 실제 입력 C 코드"다.

---

## 3. 어셈블리 파일 (`.s`)

### 이것은 무엇인가

컴파일러가 C 코드를 특정 CPU 아키텍처용 어셈블리 언어로 번역한 결과다.

생성 예:

```bash
gcc -S hello.c -o hello.s
```

### 어셈블리란 무엇인가

어셈블리는 기계어를 사람이 읽기 조금 더 좋게 적은 저수준 표현이다.

예를 들어 x86-64에서는 대략 이런 식의 명령이 보인다.

- `mov`
- `add`
- `call`
- `cmp`
- `jmp`

### 이 단계의 특징

- C의 고수준 표현이 CPU 명령 단위에 가까워진다.
- 지역 변수 접근, 함수 호출, 분기, 반환이 어떻게 바뀌는지 보인다.
- 최적화 수준에 따라 결과가 꽤 달라진다.
- 여전히 사람이 어느 정도 읽을 수 있다.

### 왜 존재하는가

컴파일러는 C 코드를 바로 기계어로 만들 수도 있지만, 개념적으로는 먼저 어셈블리 수준의 표현으로 생각하면 이해가 쉽다. `gcc -S`는 그 중간 표현을 밖으로 꺼내 보는 기능이다.

### 이 파일을 보는 이유

다음 질문에 답할 때 유용하다.

- 함수 호출은 실제로 어떻게 되는가?
- 지역 변수는 스택에서 어떻게 다뤄지는가?
- `if`, `for`, `while`은 분기로 어떻게 바뀌는가?
- 최적화가 어떤 코드를 없애거나 합치는가?

### 직관적으로 이해하면

`.s`는 "CPU가 이해하기 직전 단계의 사람이 읽을 수 있는 저수준 코드"다.

---

## 4. 오브젝트 파일 (`.o`)

### 이것은 무엇인가

어셈블리 코드를 기계어와 메타데이터로 바꾼 결과다.

생성 예:

```bash
gcc -c hello.c -o hello.o
```

### 중요한 점

이 파일은 **기계어를 포함하지만 보통 아직 실행 파일은 아니다.**

### 이 안에 들어 있는 것

대표적으로 아래 정보가 들어 있다.

- 함수의 기계어 코드
- 데이터 영역 정보
- 심볼(symbol) 정보
- 재배치(relocation) 정보
- 섹션(section) 정보

### 심볼이란 무엇인가

심볼은 함수명, 전역 변수명 같은 "링커가 나중에 찾아 연결해야 하는 이름"이라고 보면 된다.

예:

- 이 파일 안에서 정의된 함수
- 다른 파일이나 라이브러리에서 가져와야 할 함수 (`printf` 같은 것)

### 재배치 정보란 무엇인가

아직 정확한 주소를 모르는 상태에서, "나중에 링크할 때 이 위치를 수정해야 함"을 기록한 정보다.

### 왜 실행 파일이 아닌가

오브젝트 파일은 보통 프로그램의 일부 조각이다.

예를 들어 `main.o`는 `helper.o`와 결합해야 할 수도 있고, `printf`는 libc와 연결해야 할 수도 있다. 이런 연결이 아직 끝나지 않았기 때문에 바로 실행 가능한 완성품이 아니다.

### 직관적으로 이해하면

`.o`는 "기계어로 된 부품"이다.

---

## 5. 링크 결과물

링커(linker)는 여러 오브젝트 파일과 라이브러리를 연결해서 최종 산출물을 만든다.

이 산출물은 보통 아래 중 하나다.

- 실행 파일
- 정적 라이브러리 (`.a`)
- 공유 라이브러리 (`.so`)

---

## 6. 최종 실행 파일

### 이것은 무엇인가

운영체제가 로드해서 실행할 수 있는 완성된 프로그램 파일이다.

생성 예:

```bash
gcc hello.o -o hello
```

또는 보통 한 번에:

```bash
gcc hello.c -o hello
```

### 이 안에 있는 것

- 프로그램 코드
- 데이터
- 진입에 필요한 메타데이터
- 어떤 라이브러리가 필요한지에 대한 정보
- 운영체제가 읽을 수 있는 파일 형식 정보

Linux에서는 보통 ELF 형식이다.

### 왜 중요한가

운영체제는 이 파일을 보고:

- 프로세스를 만들고
- 코드/데이터를 메모리에 매핑하고
- 동적 라이브러리를 연결하고
- 시작 코드를 실행한 뒤
- `main`으로 이어지게 한다.

### 직관적으로 이해하면

실행 파일은 "OS가 바로 실행할 수 있는 완성된 제품"이다.

---

## 7. 정적 라이브러리와 공유 라이브러리

최종 링크 결과는 실행 파일만이 아니다.

### 정적 라이브러리 (`.a`)

- 오브젝트 파일 묶음에 가깝다.
- 링크 시점에 필요한 코드가 실행 파일 안으로 복사되어 들어간다.
- 배포는 단순해질 수 있다.
- 대신 최종 바이너리가 커질 수 있다.

### 공유 라이브러리 (`.so`)

- 실행 파일 바깥에 두고 실행 시점에 로드한다.
- 여러 프로그램이 같은 라이브러리를 공유할 수 있다.
- 배포와 실행 환경 의존성을 더 신경 써야 한다.

이 둘은 토픽 2에서 더 깊게 다루게 된다. 지금은 "링크 결과가 실행 파일 하나만은 아니다" 정도를 기억하면 된다.

---

## 8. 각 단계에서 사람이 볼 수 있는 것과 볼 수 없는 것

### `.c`

- 사람이 가장 잘 읽을 수 있다.
- 의도와 구조가 잘 보인다.
- 실제 기계 수준 동작은 아직 숨겨져 있다.

### `.i`

- 전처리 결과를 볼 수 있다.
- 매크로와 include가 실제로 어떻게 펼쳐졌는지 보인다.
- 여전히 논리 구조는 남아 있다.

### `.s`

- CPU에 가까운 제어 흐름이 보인다.
- 함수 호출, 분기, 스택 조작이 드러난다.
- 타입 개념은 대부분 사라진다.

### `.o`

- 기계어 수준 부품이다.
- 사람이 바로 읽기 어렵다.
- `objdump`, `readelf`, `nm` 같은 도구로 조사한다.

### 실행 파일

- OS가 실행 가능한 완성품이다.
- 내부 구조는 도구 없이는 보기 어렵다.
- 실행과 배포의 단위가 된다.

---

## 9. 왜 이 구분이 중요한가

이걸 구분할 줄 알아야 아래 문제가 각각 어디에서 생기는지 감이 생긴다.

- 매크로 문제: 전처리 단계
- 타입/문법 문제: 컴파일 단계
- 아키텍처별 코드 생성 문제: 어셈블리/컴파일 단계
- 심볼 누락 문제: 링크 단계
- 실행 시 라이브러리 누락 문제: 로딩/실행 단계

즉, "에러가 어디서 발생했는가"를 제대로 이해하려면 파일 단계별 의미를 알아야 한다.

## 10. `01_array_pointer.c`에 적용해서 보면

예를 들어 [01_array_pointer.c](D:\DEV\developer\Study\Language\C\01_c_실행_모델과_특별한_문법\exercises\01_array_pointer.c)를 기준으로 보면:

### `.c`

- 네가 작성한 원본 코드
- 배열과 포인터 실험 의도가 그대로 보임

### `.i`

- `#include <stdio.h>`가 펼쳐짐
- 표준 라이브러리 선언이 들어감

### `.s`

- `printf` 호출이 `call` 같은 어셈블리 호출로 표현됨
- `arr + 1`, `ptr + 1`이 실제 주소 계산으로 바뀜

### `.o`

- `main`, `inspect_array_parameter`의 기계어가 들어 있음
- `printf`는 아직 외부 심볼일 수 있음

### 최종 실행 파일

- libc와 연결된 실제 실행 가능한 프로그램
- `./01_array_pointer`로 실행 가능

## 11. 지금 단계에서 꼭 잡아둘 최소 개념

1. `.c`는 사람이 쓰는 원본 코드다.
2. `.i`는 전처리가 끝난 실제 컴파일 입력이다.
3. `.s`는 어셈블리 코드다.
4. `.o`는 기계어를 담은 부품이다.
5. 실행 파일은 링크가 끝난 완성품이다.
6. 링크는 여러 부품과 라이브러리를 묶는 과정이다.
7. 실행은 소스 코드가 아니라 최종 실행 파일을 대상으로 일어난다.

## 12. 직접 해볼 추천 명령

아래를 같은 파일로 직접 해 보면 감각이 금방 생긴다.

```bash
gcc -E 01_array_pointer.c -o 01_array_pointer.i
gcc -S 01_array_pointer.c -o 01_array_pointer.s
gcc -c 01_array_pointer.c -o 01_array_pointer.o
gcc 01_array_pointer.c -o 01_array_pointer
./01_array_pointer
```

그 다음 아래도 해 보면 좋다.

```bash
ls -lh
file 01_array_pointer.o
file 01_array_pointer
nm 01_array_pointer.o
objdump -d 01_array_pointer.o | less
```

이 과정을 한 번 직접 보면, "C가 왜 복잡해 보이는지"가 오히려 덜 막연해진다.  
복잡한 것이 아니라, 중간 단계가 더 많이 보이는 것이다.
